# PR 提交与合并约束

本项目所有代码变更都应基于 Pull Request 提交。每次提交代码到仓库前，必须遵守以下规则。

## 基本原则

1. **基于 PR 添加新功能**
   新功能、功能修改、重构、修复都应通过 PR 提交，避免直接在主分支上提交代码。

2. **每个 PR 只做一件事**
   每个 PR 只实现或修改单一功能。鼓励尽可能小、粒度尽可能细的 PR，便于 review、测试和回滚。

3. **大功能拆分提交**
   对于较大的功能，应拆分为多个独立 PR 分步提交。例如：

   - PR 1：新增数据模型和接口契约。
   - PR 2：新增后端服务和 API。
   - PR 3：新增前端页面和交互。
   - PR 4：接入真实数据和测试。

4. **主分支始终可运行**
   PR 合并后，主分支代码必须保持可运行状态。不得合并会导致前端无法构建、后端无法启动或核心页面不可访问的代码。

## PR 标题与描述要求

每个 PR 的标题与描述必须清晰完整，至少包含以下内容。

### 标题

用一句话说明本 PR 新增或修改了什么。

示例：

```text
新增小说 TXT 上传与规则分章接口
```

### 功能描述

说明该功能的作用与使用方式。

需要回答：

- 这个 PR 解决什么问题？
- 用户或开发者如何使用？
- 是否影响已有功能？

### 实现思路

简要说明技术选型或核心实现逻辑。

需要说明：

- 改动涉及哪些模块？
- 核心流程是什么？
- 是否引入新依赖？
- 是否有后续扩展点？

### 测试方式

说明如何验证该功能正常运行。

至少包含：

- 执行过的命令。
- 手动验证步骤。
- 需要重点关注的页面、接口或边界情况。

示例：

```text
npm run web:build
python -m compileall app

手动验证：
1. 打开 http://localhost:5173/import。
2. 上传 TXT 文件。
3. 确认页面展示章节拆分结果。
```

## 合并前检查清单

合并 PR 前，请确认：

- PR 只包含一个明确功能或修改目标。
- 标题能准确概括变更。
- 描述包含功能描述、实现思路和测试方式。
- 前端构建通过。
- 后端基础检查通过。
- 主分支合并后仍可运行。
- 无无关格式化、临时文件、调试日志或未使用代码。
