# 小说转剧本创作工作台项目介绍

## 1. 需求背景

长篇小说影视化改编的难点，不是简单把文字改成对白，而是要完成一整套前期开发工作：读懂故事结构、梳理人物关系、提炼关键事件、拆分可拍场景、判断戏剧冲突，再把这些素材转化成剧本、分镜和视频生成任务。

传统流程高度依赖人工阅读和表格整理，常见问题包括：

- 长文本信息量大，人物、事件、地点、时间线容易遗漏。
- 小说叙事和剧本表达不同，直接让大模型“一次生成完整剧本”容易失控、编造或丢失原文依据。
- 改编过程需要反复校对原文，但普通生成工具缺少来源追溯。
- 从小说到剧本、分镜、视频之间缺少统一的结构化数据链路。

本项目的目标是做一个面向影视改编前期的 AI 创作工作台。它不是追求“一键生成最终剧本”，而是把小说拆解成可审阅、可编辑、可追溯、可继续生成的中间结构，让人和 AI 共同完成改编。

## 2. 产品定位

项目定位为“小说影视化改编的结构化工作台”。核心路径是：

```text
小说导入
  -> 自动分章
  -> 叙事信息抽取
  -> 人物/关系/事件/场景可视化
  -> 场景级剧本生成与编辑
  -> 分镜图提示词与图片生成
  -> 视频生成任务管理
```

这个定位的重点是把 AI 从“直接写成品”的黑箱，变成“逐层整理素材、辅助创作决策”的协作者。

## 3. 核心功能

### 3.1 小说导入与叙事分析

- 支持 TXT、Markdown、Docx 文档导入。
- 后端自动解析文本并按章节标题切分。
- 用户可在页面配置 DeepSeek API Key。
- 叙事分析以异步任务运行，支持启动、查询状态、失败重试和重新分析。
- 每章抽取角色、事件、地点、时间、关系、冲突、对白、动作、动机、因果链、环境、分镜规划、总场景和子场景。

### 3.2 角色、关系与时间线

- 角色管理页展示人物卡片、别名、重要性、出场章节、角色功能和原文证据。
- 关系图展示人物之间的同盟、敌对、亲属、上下级、暧昧等关系及强度。
- 时间线按章节组织事件、地点、人物、冲突和结果变化。
- 多个页面支持点击来源证据，回到小说原文进行校对。

### 3.3 场景拆分与剧本生成

- 场景页采用“总场景 -> 子场景 -> 事件/对白/环境/分镜”的层级结构。
- 子场景更接近剧本 scene，是后续生成剧本正文的最小创作单元。
- 剧本页支持选择单个场景，基于结构化素材调用 AI 补全剧本正文。
- 剧本草稿保存在本地，支持继续人工编辑和 YAML 导出。
- YAML Schema 为后续剧本校验、版本管理和跨工具流转提供契约。

### 3.4 分镜图片与视频任务

- 系统从场景、镜头规划和剧本文本中生成分镜图片提示词。
- 支持调用 Seedream 生成分镜参考图。
- 支持将小说、章节、场景、镜头或分镜图片导入视频生成任务。
- 视频任务页可配置 Seedance 模型、比例、时长、清晰度、Seed、固定镜头等参数。
- 媒体管理页保留图片、视频与小说/章节/场景/镜头的关联标签，方便回溯。

## 4. 核心代码架构

项目采用 monorepo 结构，前后端分离，领域模型和接口契约清晰。

```text
apps/
  web/          React + TypeScript 前端工作台
  api/          FastAPI 后端服务
packages/
  shared/       前后端共享类型
schemas/
  screenplay.schema.yaml  剧本 YAML 结构契约
docs/
  架构、抽取、剧本生成、视频生成和 PR 规范文档
```

### 4.1 前端架构

前端按功能域组织：

```text
apps/web/src/
  app/          路由、全局布局、应用壳
  features/     导入、角色、关系、时间线、场景、剧本、图片、视频等页面
  shared/       API 映射、通用类型、当前小说状态、来源追溯组件
```

前端职责主要是把结构化结果变成创作工作流：

- 导入页负责文档上传、模型配置、分析状态展示。
- 角色页和关系页负责人物理解。
- 时间线页负责事件顺序和冲突变化。
- 场景页负责把小说叙事整理成可拍摄单元。
- 剧本页负责场景级生成、编辑和导出。
- 图片/视频页负责把剧本素材继续转化为视听生成任务。

### 4.2 后端架构

后端采用 FastAPI + Pydantic，按层拆分：

```text
apps/api/app/
  api/           HTTP 路由
  core/          配置、日志、存储路径、模型服务配置
  domain/        Pydantic 领域模型
  services/      文档解析、章节分析、剧本补全、图片/视频服务
  repositories/  本地文档与工作区存储
```

路由层只负责接收请求和返回响应，核心业务逻辑放在 service 中。这样后续替换存储、模型或新增队列时，不需要大幅改动前端和接口。

### 4.3 核心后端模块

- `document_parser.py`：读取 TXT/Markdown/Docx，规范化文本并切分章节。
- `workspace_service.py`：管理文档导入、分析任务状态、分析结果保存和恢复。
- `chapter_analysis_service.py`：信息抽取主流程，按章节调用 DeepSeek，解析 JSON，合并角色，关联事件和场景，补齐原文位置。
- `deepseek_client.py`：统一封装 DeepSeek 请求、JSON 解析、错误处理和调试文件输出。
- `screenplay_generation_service.py`：基于场景素材补全剧本正文。
- `storyboard_prompt_service.py`：生成角色图和分镜图提示词。
- `seedream_image_client.py` / `seedance_client.py`：对接图片和视频生成服务。

### 4.4 关键领域模型

后端领域模型围绕“从小说到剧本”的改编链路设计：

- `Chapter`：章节文本、摘要、字数、冲突提示、原文位置。
- `Character`：角色、别名、重要性、角色功能、出场章节、来源证据。
- `Event`：事件标题、摘要、参与人物、地点、时间、冲突、结果。
- `NarrativeBlock`：总场景，承载一个章节或连续章节的大段叙事目标。
- `SubScene`：子场景，更接近可拍摄、可生成剧本的 scene。
- `Dialogue` / `Action` / `EnvironmentInfo` / `ShotPlan`：剧本和分镜生成的素材层。
- `SourceRef`：所有关键对象的原文来源引用。

## 5. 为什么这样抽取小说信息

### 5.1 不按“整本小说一次生成”

长篇小说上下文太长，直接一次生成完整剧本会带来三个风险：

- 成本高，容易超过模型上下文限制。
- 输出不可控，结构、篇幅和格式难稳定。
- 很难定位每段剧本来自哪段原文。

所以项目选择“按章节综合抽取”，每章只调用一次模型，返回一份结构化 JSON。这样既减少 API 调用次数，又让模型在同一上下文里同时判断人物、事件、地点、冲突、对白和场景关系。

### 5.2 不只抽人物和事件

如果只抽人物和事件，后续很难生成真正可拍的剧本。剧本需要的是镜头、环境、动作、对白、冲突、情绪和场景功能。因此当前抽取对象扩展为：

```text
角色、地点、时间、事件、关系、冲突、对白、动作、动机、因果链、
环境、分镜规划、总场景、子场景、剧本场景候选、情绪曲线
```

这些字段分别服务于不同环节：

- 角色和关系用于人物卡、关系图、角色声音和人物弧光。
- 事件、冲突和因果链用于时间线、场景排序和戏剧推进。
- 环境、动作、对白用于剧本 beat 生成。
- 分镜规划用于图片提示词和视频镜头描述。
- 总场景和子场景用于把小说章节转换为可拍摄结构。

### 5.3 使用“总场景 -> 子场景”的层级结构

小说章节不等于剧本场景。一个章节可能包含多个地点、多个时间段和多个戏剧冲突；反过来，多个章节也可能构成同一个连续大场面。

因此项目引入两级结构：

```text
NarrativeBlock 总场景：理解这一章或几章整体在讲什么
SubScene 子场景：按同一地点、同一时间段、同一冲突目标拆成可拍单元
```

好处是：

- 更符合影视改编的实际工作方式。
- 用户可以先确认大段落结构，再细修每个子场景。
- 剧本生成可以按子场景逐个补全，减少模型失控。
- 分镜和视频任务可以天然绑定到更细的场景粒度。

### 5.4 强制保留 SourceRef

每个关键对象都尽量保留 `source_refs`：

```json
{
  "chapter_id": "chapter-1",
  "start_char": 120,
  "end_char": 168,
  "evidence": "原文中的连续短句"
}
```

模型只需要稳定提供 evidence，后端再用 evidence 在章节正文中查找并补齐字符位置。

这样做的价值是：

- 用户可以判断 AI 抽取得是否可信。
- 角色、事件、场景、剧本都能回到原文校对。
- 后续生成内容如果偏离原文，可以快速发现。
- 对比赛演示来说，可追溯性是区别于普通聊天式生成工具的重要亮点。

## 6. 工程质量设计

### 6.1 清晰的边界

项目把“导入、抽取、展示、生成、媒体任务”分成独立模块。前端页面不直接写后端抽取逻辑，后端路由也不直接调用模型细节，而是通过 service 层统一封装。

### 6.2 缓存与调试

章节分析使用内容 hash 缓存：

```text
schema version + chapter id + title + chapter text -> cache key
```

章节内容不变时直接读取缓存，避免重复消耗模型调用。每次模型请求还会输出调试文件，包括 system prompt、user prompt、request、raw response、parsed response 和错误信息，方便排查 prompt 和模型输出问题。

### 6.3 异步与并发控制

叙事分析按章节异步执行，并通过最大并发数控制请求节奏，避免一次性打满模型接口，也能让长篇小说分析更快完成。

### 6.4 本地优先与安全

项目以本地工作台为主：

- API Key 保存到本地 `.env`，不提交仓库。
- 模型缓存、调试文件、生成媒体文件都在本地目录，并被 `.gitignore` 忽略。
- 剧本草稿、图片任务和视频任务保留本地快照，方便演示和迭代。

### 6.5 PR 与开发规范

文档中规定了 PR 拆分、中文 Conventional Commits、构建检查和 PR 描述模板。大功能建议按“模型/接口/前端/真实数据/测试”拆分，保证主分支持续可运行。这一点对应比赛评分中的开发过程与质量。

## 7. 创新点

1. 从“一次性生成剧本”转为“结构化改编工作台”  
   重点不是让 AI 替代创作者，而是让 AI 把复杂小说整理成创作者能审阅和继续加工的素材。

2. 面向影视改编的信息抽取 schema  
   抽取的不只是 NLP 常见实体，而是事件、冲突、对白、动作、环境、镜头、场景功能等可直接服务剧本和分镜的素材。

3. 总场景与子场景的层级建模  
   将小说叙事结构转换为影视创作结构，让后续剧本、分镜和视频生成都有稳定粒度。

4. 全链路来源追溯  
   从角色卡、事件、场景到剧本生成，都能回到原文证据，降低模型编造风险。

5. 小说到视频的连续工作流  
   项目不止停在剧本生成，还继续连接分镜图片和视频任务，让“文字 IP 影视化”形成可演示闭环。
