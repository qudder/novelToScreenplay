# 小说转剧本创作工作台评审问答参考

本文档用于比赛答辩准备。回答不是唯一标准答案，可以根据现场演示进度压缩表达。建议答辩时先回答结论，再补一句项目里的实现依据。

## 一、产品与需求

### 1. 你的项目解决的是“小说转剧本”的哪一个核心痛点？为什么不是直接让大模型生成完整剧本？

核心痛点是长篇小说改编前期的信息整理成本高，而且直接生成完整剧本不可控。小说里有大量人物、事件、地点、时间、对白和心理描写，编剧需要先把它们整理成可拍摄的场景素材。  
所以本项目不是直接让模型写一篇成品，而是先把小说拆成结构化数据，再逐步生成剧本和分镜。这样结果可检查、可编辑、可追溯，也更符合真实改编流程。

### 2. 目标用户是谁？

主要用户是影视改编前期人员，包括编剧、短剧团队、AI 视频创作者、网文 IP 运营方和内容策划。  
其中短剧团队和 AI 视频创作者最适合早期版本，因为他们需要快速把小说内容拆成场景、分镜和视频任务，强调效率和可视化。

### 3. 如果用户已经会用 ChatGPT/DeepSeek 写剧本，为什么还需要你的工作台？

通用聊天工具擅长生成文本，但缺少项目级工作流。它不会稳定保存人物、事件、场景、来源证据、分镜和视频任务之间的关系。  
本项目提供的是从小说导入、结构化抽取、原文追溯、场景拆分、剧本编辑、分镜生成到视频任务管理的一条链路。用户不是只拿到一段生成文本，而是得到一套可继续加工的创作资料库。

### 4. 你的产品更像写作工具、剧本管理工具，还是影视前期开发工具？

更接近影视前期开发工具。写作只是其中一环，项目更关注把小说 IP 变成可用于剧本、分镜和视频生成的结构化资产。  
它既能辅助剧本创作，也能管理角色、事件、场景和视听素材。

### 5. 当前流程中，哪些环节必须人工参与？哪些环节可以自动化？

自动化部分包括文档分章、信息抽取、初步场景拆分、剧本补全、分镜提示词生成和媒体任务创建。  
人工必须参与的是确认抽取是否准确、合并或删除不合适的场景、判断改编取舍、修改台词和最终审稿。项目定位是人机协作，不是完全替代创作者。

### 6. 你如何判断一部小说是否适合被拆成剧本？

主要看是否有清晰的人物目标、冲突推进、事件变化和可视化动作。项目会抽取事件、冲突、动作、对白、环境和情绪曲线，这些数据越丰富，越适合剧本化。  
如果一部小说大量依赖内心独白或散文式表达，系统也能整理信息，但需要更多人工改写成外部动作和对白。

### 7. 长篇小说、短篇小说、短剧爽文、严肃文学，对你的流程有什么不同影响？

长篇小说更需要章节级缓存、并发分析和跨章节场景合并。短篇小说可以更快形成完整剧本草稿。  
短剧爽文通常事件密度高、冲突明确，适合快速拆成场景。严肃文学可能心理描写更多，需要更强调动作化改写和人工审校。

## 二、信息抽取设计

### 8. 为什么按章节抽取，而不是按段落、按整本、或者按固定 token 窗口抽取？

章节是小说天然的叙事单元，通常包含相对完整的事件推进和人物变化。  
按整本抽取上下文太长且成本高；按段落太碎，容易丢失事件上下文；固定 token 窗口会切断章节和场景。按章节抽取在成本、稳定性和叙事完整性之间比较平衡。

### 9. 哪些抽取字段是剧本生成最关键的？

最关键的是角色、事件、冲突、对白、动作、环境、场景和来源引用。  
角色决定谁在场，事件和冲突决定戏剧推进，对白和动作决定剧本 beat，环境决定场面调度，SourceRef 保证能回到原文校对。

### 10. 如果模型抽错人物关系，系统怎么让用户发现并修正？

关系图会把人物关系可视化，角色卡和事件卡也会展示原文证据。用户可以通过 SourceRef 回到原文检查模型判断是否准确。  
当前版本主要强调发现和校对，后续可以增加关系编辑、版本记录和人工确认状态。

### 11. 如何处理同一个角色的别名、称谓、代词和身份变化？

抽取时要求模型输出 aliases，后端按角色主名做初步合并，并合并别名、出场章节和来源证据。  
对于复杂别名和身份变化，当前仍需要人工确认。后续可以引入别名合并规则、角色知识库和跨章节一致性检查。

### 12. 如何避免把地名、组织名、物品名误识别成角色？

Prompt 中明确要求不要把地点、机构、朝代、抽象概念和物品识别为角色。后端解析时也要求角色必须有 name。  
更重要的是前端展示来源证据，用户可以快速发现误识别。后续可以结合词典、实体类型校验和人工确认来进一步降低误判。

### 13. 事件和场景的区别是什么？为什么需要两者都抽？

事件是“发生了什么”，强调目标、阻力和结果变化；场景是“如何拍出来”，强调地点、时间、出场人物和戏剧功能。  
一个场景可以包含多个事件，一个事件也可能被压缩成一个动作 beat。两者都抽，才能既保持叙事逻辑，又服务剧本和分镜生成。

### 14. 为什么引入“总场景”和“子场景”？它相比平铺场景列表有什么优势？

总场景承载一章或多章的整体叙事目标，子场景承载更细的可拍摄单元。  
平铺列表适合展示信息，但不符合影视改编的分层工作方式。层级结构能让用户先确认大段落，再编辑具体场景，也便于后续按子场景生成剧本、分镜和视频任务。

### 15. SourceRef 的作用是什么？如果 evidence 在原文中匹配不到怎么办？

SourceRef 是原文追溯结构，包含 chapter_id、start_char、end_char 和 evidence。它让角色、事件、场景和剧本都能回到小说依据。  
如果 evidence 匹配不到，系统保留 start_char/end_char 为 -1，前端仍能显示章节和证据文本。后续可以用模糊匹配或向量检索提升定位成功率。

### 16. 如何处理跨章节事件？

当前先按章节抽取，再通过 NarrativeBlock、CausalLink、相邻章节地点和人物重合等信息做跨章节组织。  
第一版可默认一章一个总场景，后续逐步支持相邻章节自动合并，把连续冲突或连续行动归入同一个总场景。

### 17. 如何判断哪些对白值得保留？

Prompt 要求只保留推动冲突、揭示关系、改变局面、制造悬念或有角色声音的关键对白。  
普通寒暄或说明性弱的对白不优先抽取，这样后续剧本生成更聚焦。

### 18. 分镜规划为什么要在信息抽取阶段就做，而不是等剧本生成之后再做？

小说原文里有很多动作、视线、空间关系和情绪变化，适合在抽取阶段就转成镜头素材。  
这样剧本生成时可以参考镜头调度，分镜和视频任务也能直接复用这些素材，形成从小说到视听表达的连续链路。

## 三、AI 与可靠性

### 19. 你如何控制大模型幻觉？

主要有三层控制：第一，要求模型输出结构化 JSON，而不是自由发挥；第二，强制关键对象提供 source_refs；第三，前端展示原文证据供用户校对。  
剧本生成阶段也尽量按单场景生成，减少模型一次性发挥过多。

### 20. 模型返回 JSON 格式错误时系统怎么处理？

DeepSeek 客户端会先尝试去掉代码块，再从文本中提取 JSON object。如果仍然解析失败，会记录 parse_error、原始响应和调试文件，并让该章分析失败或返回空结果。  
这样问题可定位，不会悄悄污染后续数据。

### 21. 为什么使用 DeepSeek？是否可以替换为其他模型？

DeepSeek 在中文长文本理解和成本方面比较适合这个原型。  
架构上模型调用封装在 deepseek_client 和相关 service 中，后续可以替换为 OpenAI、豆包、通义或本地模型，只要保持结构化 JSON 输出契约即可。

### 22. 当前 prompt 的核心约束是什么？

核心约束是只输出合法 JSON object，不输出 Markdown 或解释；抽取面向剧本生成的结构化数据；不要编造当前章节没有的信息；关键对象必须带 source_refs；证据文本要尽量来自原文短句。  
同时限制每类对象数量，避免输出过长和失控。

### 23. 如何评估抽取质量？有没有人工标注或对比指标？

当前原型主要通过人工检查和原文追溯验证。可以从角色召回率、事件准确率、source_refs 定位率、场景可用率、剧本生成可编辑程度几个维度评估。  
后续可以建立小规模人工标注集，对比模型抽取结果和人工整理结果。

### 24. 如果章节太长超过上下文限制，你准备怎么处理？

可以在章节内部再按自然段或场景线索切分，但保留 chapter_id 和局部 offset。  
处理策略是“章节优先，超长再分块”，避免一开始就把所有文本切得太碎。

### 25. 缓存策略如何避免旧 prompt 产生的旧结构污染新结果？

缓存 key 中包含 CACHE_SCHEMA_VERSION。只要 prompt 或 schema 发生重要变化，就提升版本号，让旧缓存自动失效。  
同时提供重新分析入口，必要时强制刷新缓存。

### 26. 如果同一章节重新分析结果不一致，会不会影响后续剧本？

会有影响，所以系统通过缓存保证同一内容默认复用同一结果。重新分析是用户主动触发的操作。  
后续可以引入分析结果版本管理，让用户比较不同版本并选择保留。

### 27. 如何平衡抽取字段很多导致 prompt 复杂、输出不稳定的问题？

当前做法是限制每类字段数量，并要求所有对象字段固定存在。  
长期可以把抽取拆成“基础叙事抽取”和“分镜/剧本素材增强”两层，或者对重要章节做二次精抽。

## 四、剧本生成

### 28. 小说改编成剧本时，哪些内容应该保留，哪些内容应该压缩？

应该保留推动剧情的行动、冲突、关键对白、人物关系变化和场面氛围。  
大量背景说明、重复心理描写和弱动作叙述可以压缩成 action beat、视觉细节或改编说明。

### 29. 系统如何把小说叙述转换成可拍动作？

抽取阶段会提取 Action、EnvironmentInfo、Dialogue 和 ShotPlan。生成剧本时优先使用这些外部可见素材，把心理活动转化为表情、动作、空间调度或对白。  
Prompt 也会要求动作 beat 只写可拍内容，避免堆心理描写。

### 30. 生成剧本时，如何保证角色说话风格一致？

角色的 role、description、历史 Dialogue、Motivation 和后续 screenplay voice_note 都可以作为约束。  
当前主要依赖场景上下文和关键对白样本，后续会把角色声音整理成稳定的角色小传和台词风格说明。

### 31. 剧本生成为什么按单场景生成，而不是一次生成整集/整部？

单场景生成更可控，输入上下文更小，输出更容易编辑，也更方便缓存和重试。  
如果整集一次生成，模型容易丢事件、乱顺序或编造内容。

### 32. YAML Schema 在项目中起什么作用？

YAML Schema 是剧本草稿的结构契约，定义剧本元信息、角色、幕、场景、beats、来源和校验字段。  
它让导出结果不是一段纯文本，而是可被继续编辑、校验、版本管理和接入其他工具的结构化剧本。

### 33. 用户编辑后的剧本如何和原始抽取结果保持关联？

剧本场景和 beat 会尽量保留 source_event_ids、source_span 或 SourceRef。  
即使用户修改文本，仍可以知道这个场景最初来自哪些章节和事件。后续可以增加编辑历史和变更说明。

### 34. 如果用户修改了场景顺序，后续分镜和视频任务如何同步？

场景、分镜图和视频任务都有 id 和关联标签。修改顺序时应更新剧本草稿中的 sequence_order，但保留原有 id，这样已有分镜和视频任务仍能回链到正确场景。  
后续可以增加“同步状态”提示，提醒用户哪些媒体任务基于旧版本生成。

### 35. 如何处理模型新增剧情？是否会标注“为影视化新增”？

设计上会通过 adaptation 字段或 revision_note 标注新增、压缩、合并和改写原因。  
这样用户能区分哪些来自原文，哪些是为了影视化补充的内容。

## 五、前端交互

### 36. 用户从导入小说到生成第一段剧本，需要经过多少步？

主要是四步：上传小说、启动叙事分析、选择一个子场景、点击 AI 补全剧本。  
中间角色、关系和时间线不是强制步骤，但可以帮助用户检查分析质量。

### 37. 哪个页面是用户最核心的工作台？为什么？

场景页和剧本页是核心工作台。场景页负责把小说变成可拍摄结构，剧本页负责把结构化素材变成可编辑剧本文本。  
这两个页面连接了“理解小说”和“创作剧本”。

### 38. 角色页、时间线页、场景页之间如何形成工作流，而不是孤立展示？

角色页帮助理解人物，时间线页帮助理解事件顺序，场景页把人物和事件组织成可拍单元。  
它们共享同一份分析结果和 SourceRef，用户可以从任意页面回到原文校对。

### 39. 原文对照弹窗解决了什么问题？

它解决的是可信度问题。用户不需要盲信模型结果，可以直接看到角色、事件或场景对应的原文证据。  
这对改编类产品很重要，因为原文依据决定了后续生成是否靠谱。

### 40. 如果抽取结果很多，如何避免页面信息过载？

通过分层展示和按场景聚合来降低信息压力。用户先看总场景，再展开子场景，再看事件、对白、环境和镜头。  
此外可以通过筛选、折叠、标签和重要性排序来控制信息密度。

### 41. 是否支持用户手动修改角色、事件、场景？修改后如何保存？

当前重点是展示、生成和本地草稿保存，部分页面已有本地状态和快照能力。  
完整的人工编辑和持久化是下一步重点，包括角色合并、事件修正、场景调整和版本保存。

### 42. 当前交互里最有创意或最能体现产品价值的是哪一处？

最能体现价值的是场景页的层级结构和原文追溯，以及从子场景直接生成剧本、分镜和视频任务的链路。  
它把“小说理解”和“影视生成”接到了一起。

## 六、后端与架构

### 43. 为什么采用 React + FastAPI？

React + TypeScript 适合构建复杂交互工作台，便于做多页面状态、图谱、编辑器和媒体任务管理。  
FastAPI + Pydantic 适合快速搭建结构化 API，类型清晰，也方便处理异步模型请求。

### 44. 后端为什么分 api/core/domain/services/repositories 这些层？

这是为了保持职责清晰。api 只处理 HTTP，domain 定义数据模型，services 放业务逻辑，repositories 处理存储，core 放配置和通用能力。  
这样后续替换数据库、模型服务或任务队列时，不会牵一发动全身。

### 45. 当前是本地存储，如果要多人协作，哪些模块需要替换？

主要替换 repositories 和部分 workspace 状态管理。可以把本地 document_store 换成 PostgreSQL，把人物关系和事件网络接入 Neo4j，把媒体文件接入对象存储。  
前端和大部分 service 逻辑可以尽量复用。

### 46. memory repository 和 document store 的边界是什么？

memory repository 更像早期工作区数据的内存容器，document store 负责本地文档记录、导入结果和分析结果的保存。  
后续可以统一抽象为项目仓库，支持持久化和多用户。

### 47. 分析任务为什么用异步后台任务？如果服务重启任务会怎样？

叙事分析涉及多章节模型调用，耗时较长，不能阻塞 HTTP 请求，所以用后台任务并通过状态轮询。  
当前原型中服务重启会中断正在运行的任务，这是后续需要用任务队列和持久化任务状态改进的地方。

### 48. 章节并发分析的最大并发数怎么确定？

需要在速度、模型接口限流和稳定性之间平衡。当前通过配置控制最大并发，默认较保守。  
如果模型服务限流更宽，可以提高并发；如果失败率增加，则降低并发。

### 49. 生成媒体文件为什么放在 `.data/generated_media`？

生成图片和视频属于本地产物，体积大、可重复生成，不应该提交到仓库。  
放在 `.data/generated_media` 便于统一管理、前端访问和 gitignore 忽略。

### 50. 如何保护 API Key 和用户上传的小说原文？

API Key 保存在本地 `.env`，不提交仓库，日志也不打印完整 key。  
小说原文和调试文件当前以本地优先方式保存，适合个人工作台。商业化时需要增加加密存储、权限控制和数据清理策略。

## 七、工程质量

### 51. 做了哪些设计来保证代码可维护？

前后端分离、后端分层、领域模型集中定义、模型调用统一封装、前端按 feature 组织。  
同时有 docs 文档记录架构、抽取、剧本生成、视频生成和 PR 规范，方便持续迭代。

### 52. TypeScript 类型、Pydantic 模型和 YAML Schema 之间如何保持一致？

Pydantic 模型约束后端接口，TypeScript 类型约束前端使用，YAML Schema 约束导出的剧本结构。  
当前主要靠文档和映射函数维护一致，后续可以引入 schema 生成工具，减少手写同步成本。

### 53. 如果以后新增“审稿 agent”，应该放在哪一层？

应该放在 services 层，作为 screenplay 或 review 相关服务。  
它可以读取 domain 模型中的角色、场景、剧本和 SourceRef，输出审稿意见、风险提示和修改建议。

### 54. 当前项目最大的技术债是什么？

主要是本地存储和任务系统还偏原型，缺少正式数据库、任务队列和完整的编辑持久化。  
另外前后端类型和 schema 还需要更自动化地保持同步。

### 55. 哪些地方最需要补测试？

最需要补的是文档解析和分章、模型 JSON 解析、SourceRef 定位、章节分析缓存、API 映射和剧本导出。  
这些地方一旦出错，会影响主链路稳定性。

### 56. 如何证明主流程是可运行的？

可以现场演示上传小说、启动分析、查看角色/事件/场景、点击原文追溯、生成单场剧本、创建分镜或视频任务。  
工程层面可以展示 `npm run web:build` 和 `python -m compileall apps/api/app` 等基础检查。

### 57. PR 拆分和 commit 规范如何体现开发过程质量？

文档要求每个 PR 只做一件事，标题和描述说明功能、实现、测试和风险，commit 使用中文 Conventional Commits。  
这说明项目不是堆功能，而是有可 review、可回滚、可持续迭代的开发过程。

### 58. 有没有日志、调试文件或错误恢复机制？

有。模型请求会保存 system prompt、user prompt、request、raw response、parsed response 和错误信息。  
章节分析有缓存和重试入口，失败章节也可以记录 empty_chapter_ids 或错误消息，方便定位问题。

## 八、创新性与差异化

### 59. 相比普通 AI 写作工具，最核心的创新是什么？

核心创新是把小说改编变成结构化、可追溯的工作流，而不是一次性文本生成。  
系统抽取的是面向影视化的素材：冲突、动作、对白、环境、镜头和场景层级，这些是普通写作工具不系统管理的。

### 60. 相比只做剧本生成，为什么继续做到分镜图和视频任务？

因为当前 AI 创作趋势是从文本到图像再到视频。小说 IP 改编不止停在剧本，还需要视觉验证和视频生成。  
分镜和视频任务让项目形成从文本理解到视听生成的闭环，更能体现完整度和创新性。

### 61. “小说 IP 到视频生成”的未来工作流会是什么样？

未来会是：小说结构化理解 -> 编剧确认 -> 场景级剧本 -> 分镜设计 -> 角色和场景视觉一致性 -> 视频生成 -> 审稿和迭代。  
本项目现在做的是这条链路的原型，把关键数据结构和交互流程先跑通。

### 62. 系统里哪些设计是为未来多 agent 协作预留的？

领域模型、SourceRef、场景层级、剧本 schema 和 service 分层都为多 agent 预留了接口。  
未来可以有抽取 agent、场景 agent、剧本 agent、分镜 agent、审稿 agent，各自处理不同阶段，但共享同一套结构化数据。

### 63. 如果评委只能记住一个亮点，你希望他们记住什么？

我希望评委记住：这个项目不是简单 AI 写剧本，而是把小说变成可追溯、可编辑、可继续生成视频的结构化创作资产。

## 九、演示与落地

### 64. 准备演示哪一本小说？为什么选它？

建议选择人物关系清楚、事件推进明显、场景变化丰富的样例，比如已有项目中的《绍宋》片段或《三体》片段。  
这样能展示人物、冲突、场景、剧本和分镜的完整链路。

### 65. 15 分钟演示中，最关键的三步是什么？

第一，上传小说并启动叙事分析；第二，展示角色、时间线、场景和原文追溯；第三，从子场景生成剧本，并继续展示分镜或视频任务。  
这三步能覆盖产品完整度、技术架构和创新闭环。

### 66. 如果现场模型调用很慢或失败，有什么备用方案？

可以提前准备已分析好的本地缓存和演示小说。现场先说明模型调用支持实时运行，但为了保证演示节奏，使用缓存结果展示完整链路。  
这也体现了系统缓存设计的实际价值。

### 67. 如何展示“抽取结果能回到原文”？

在角色卡、事件卡或场景卡上点击来源证据，打开原文对照，展示对应章节和 evidence。  
演示时可以强调：每个关键判断都不是凭空生成，而是能回到小说依据。

### 68. 如何展示从小说到剧本再到视频任务的闭环？

先选择一个子场景，展示它关联的人物、事件、环境和镜头；再生成剧本文本；然后把该场景或镜头导入分镜图片/视频生成任务。  
这样观众能看到同一个故事单元如何一路流转。

### 69. 当前项目完成度最高的部分是什么？相对薄弱的部分是什么？

完成度较高的是小说导入、章节分析、结构化抽取、前端多页面展示和从场景到剧本/媒体任务的链路。  
相对薄弱的是多人协作、完整数据库持久化、自动化测试和更精细的人工编辑闭环。

### 70. 如果再给两周，最优先补哪三个功能？

第一，完善角色、事件和场景的人工编辑与保存。第二，增强剧本 beat 结构化编辑器。第三，加入抽取质量评估和审稿 agent。  
这三个功能能明显提升真实可用性。

## 十、尖锐问题

### 71. 这是不是只是把多个大模型 API 串起来？技术壁垒在哪里？

不是。模型 API 只是能力来源，项目的关键在于改编链路和数据结构设计。  
技术壁垒体现在面向影视化的信息抽取 schema、总场景/子场景建模、SourceRef 追溯、缓存调试机制，以及小说到剧本、分镜、视频任务的统一工作流。

### 72. 如果模型抽取质量不好，整个产品是不是就不可用了？

不会完全不可用，因为系统不是黑箱自动生成，而是把抽取结果展示给用户检查和编辑。  
SourceRef 能帮助用户发现问题，缓存和重试能改善结果。后续人工编辑和审稿 agent 会进一步提升可靠性。

### 73. 用户为什么要相信 AI 抽出来的剧情结构？

用户不需要盲信。系统把每个关键结果尽量绑定原文证据，让用户可以验证。  
信任不是来自模型本身，而是来自“结构化结果 + 原文追溯 + 人工确认”的机制。

### 74. 系统是否真的能减少编剧工作量？有没有衡量方式？

能减少前期整理工作量，尤其是人物表、事件表、场景拆分、关键对白和分镜素材整理。  
可以用人工整理同一章节所需时间、抽取结果可用率、生成剧本后修改比例等指标衡量。

### 75. 生成的视频是否和小说剧情一致？如何验证？

视频一致性依赖前面的结构化链路。视频任务不是直接从小说生成，而是从场景、剧本、分镜图和 SourceRef 逐级生成。  
验证方式是检查视频任务关联的场景、镜头和剧本是否来自正确原文，后续可加入视觉审稿和剧情一致性检查。

### 76. 如果小说版权敏感，上传和缓存是否存在风险？

存在风险，所以当前定位为本地工作台，文本、缓存和媒体文件都保存在本地。  
商业化时必须加入权限控制、加密存储、数据删除、脱敏日志和版权合规机制。

### 77. 项目现在离真实商业产品还有哪些差距？

主要差距是多人协作、稳定持久化、权限管理、支付/成本控制、抽取质量评估、更多格式导出和生产级任务队列。  
但作为比赛原型，核心链路和关键创新已经能展示。

### 78. 如果现场挑一章小说上传，系统能稳定工作吗？

只要文本格式清晰、API Key 正常、模型服务可用，主流程可以运行。  
为了演示稳定性，建议准备缓存样例，同时可以现场上传短章节证明实时能力。

### 79. 设计有没有过度复杂？MVP 最小闭环是什么？

设计覆盖了完整愿景，但 MVP 很清楚：上传小说 -> 分章 -> 抽取角色/事件/场景 -> 原文追溯 -> 生成单场剧本。  
分镜图和视频任务是增强闭环，用于展示项目未来价值。

### 80. 如何用一句话说服评委这是值得继续做的项目？

这不是一个简单的 AI 写作 Demo，而是一套把小说 IP 转化为可追溯剧本资产和视听生成任务的结构化工作台，能真实降低影视改编前期成本。
